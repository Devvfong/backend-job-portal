# Backend Architecture

## Overview

The Job Portal backend follows a **layered architecture** to ensure the system is scalable, maintainable, and easy to extend.

Tech stack:

- Node.js
- Express.js
- Prisma ORM
- PostgreSQL
- JWT Authentication

---

## Request Flow

Client requests travel through several layers:

Client  
↓  
Route  
↓  
Middleware  
↓  
Controller  
↓  
Service  
↓  
Prisma ORM  
↓  
Database

---

## Layer Responsibilities

### Routes

Routes define API endpoints and attach middleware.

Example:


POST /api/v1/auth/register
GET /api/v1/jobs
POST /api/v1/jobs/create


Routes connect HTTP endpoints to controllers.

---

### Middleware

Middleware runs before controllers.

Responsibilities:

- authentication
- request validation
- authorization
- error handling

Examples:

- protect.middleware.js
- validate.middleware.js
- authorize.middleware.js

---

### Controllers

Controllers handle:

- request data
- calling services
- sending HTTP responses

Controllers should **not contain business logic**.

---

### Services

Services contain:

- business logic
- database operations
- reusable functions

Example services:

- auth.service.js
- job.service.js

---

### Prisma ORM

Prisma handles communication with PostgreSQL.

Example:


prisma.user.findUnique()
prisma.job.create()


---

## Benefits of this Architecture

- clean separation of concerns
- easier testing
- reusable logic
- scalable structure

# Database Schema

## Overview

The Job Portal database contains four main entities:

- User
- Company
- Job
- Application

---

## Relationships

User → Application (1:N)  
Job → Application (1:N)  
Company → Job (1:N)  
Company → User (1:N)

---

## Role System

Users have roles:


job_seeker
company_admin


job_seeker:
- search jobs
- apply for jobs

company_admin:
- create jobs
- manage jobs
- view applicants

---

## Application Constraint

A user cannot apply to the same job twice.


@@unique([userId, jobId])


This ensures data integrity.

---

## Indexes

Indexes improve performance:


@@index([email])
@@index([companyId])
@@index([location])


These help with queries like job search.

# Authentication Flow

Authentication uses **JWT (JSON Web Tokens)**.

---

## Login Process

1. User sends login request
2. Server verifies credentials
3. JWT token is generated
4. Token is stored in cookie
5. Token is sent with future requests

Example header:


Authorization: Bearer TOKEN


---

## Protect Middleware

The protect middleware:

1. reads the token
2. verifies the JWT
3. fetches user from database
4. attaches user to request

Example:


req.user


This allows controllers to know who is making the request.

---

## Authorization Middleware

Authorization restricts access based on roles.

Example:


authorize("company_admin")


Example route:


router.post("/create", protect, authorize("company_admin"), createJob)


Meaning:

Only company admins can create jobs.

# Roles and Authorization

## Roles

The system currently supports two roles.

job_seeker  
company_admin

---

## job_seeker

Permissions:

- search jobs
- apply to jobs
- upload resume

---

## company_admin

Permissions:

- create jobs
- manage jobs
- view applicants
- manage company profile

---

## Security Rule

Clients cannot assign roles.

The server always controls roles during registration.

Example:


role: "job_seeker"


---

## Route Protection

Example:


router.post(
"/create",
protect,
authorize("company_admin"),
createJob
)


Execution order:

1. protect (authentication)
2. authorize (role check)
3. controller
